# MCP GUI Provider Mode Design

## Goal

Make the existing MCP server usable from the translator settings UI without adding a separate top-level tool. Users should be able to choose `MCP сервер` in the same service selector where they choose Gemini, OpenRouter, local models, and other AI services.

## Placement And Coverage

The entry point is the existing `KeyManagementWidget` service combo. This widget is reused across the AI workflow surfaces, so the MCP mode must appear everywhere that widget is used:

- main EPUB translator settings;
- AI glossary generation;
- AI glossary correction;
- untranslated-fragment fixing;
- consistency checking.

- Add a synthetic provider item named `MCP сервер`.
- This item is UI-only; it is not a translation provider and must not be saved as the active translation provider for normal translation sessions.
- The item appears at the end of the service combo so existing providers keep their relative order.

The implementation should avoid per-window copies of the MCP UI. The shared `KeyManagementWidget` owns the selector behavior and card swapping so every AI window stays consistent.

## Provider Row Behavior

When a normal provider is selected:

- The current key status card remains visible.
- Existing API/provider server button behavior remains unchanged.
- Available keys, active keys, provider session behavior, and key counts work exactly as they do now.

When `MCP сервер` is selected:

- Hide the key status card with `Ключи / Доступно / Исчерпано`.
- Hide the existing provider-specific `Запустить сервер` button to avoid confusing it with the MCP server.
- Show a compact MCP control card in the same row position where the key status card was.
- The card must fit the existing provider-row height and avoid widening the settings layout.
- Key pool and active-key areas may remain below, but they should be disabled or visually irrelevant for this synthetic provider.

When the user switches back from `MCP сервер` to a normal provider:

- Hide the MCP control card.
- Restore the key status card.
- Restore provider-specific server-button visibility according to the selected provider.
- Restore key counts and key lists for that provider.

## Compact MCP Card

The compact card contains:

- Status label: `Выключен`, `Запущен`, or `Ошибка`.
- Short detail: local daemon host/port when available, otherwise `stdio + local daemon`.
- Primary action button:
  - `Запустить` when daemon is not running.
  - `Остановить` when daemon is running.
- Config button:
  - `Codex config` copies or shows the Codex config snippet generated by the existing MCP installer logic.

The card should use the existing dark theme and compact spacing. It should feel like a sibling of `keyStatusCard`, not a large new panel.

## Backend Reuse

Reuse the existing MCP package:

- `gemini_translator.mcp.client.ensure_daemon_process()` for start.
- `gemini_translator.mcp.client.load_client()` / `client.status()` for refresh.
- `client.shutdown()` for stop.
- `gemini_translator.mcp.client_install.build_config_snippet("codex")` for config text.

Do not create a second daemon implementation. The GUI should control the same state directory and daemon used by the CLI/MCP stdio server.

## Threading And Responsiveness

Starting, stopping, and checking daemon status must not block the Qt UI thread. Use a small worker object/thread or Qt timer-driven background call.

The UI should poll lightly while the MCP card is visible:

- Refresh status when `MCP сервер` is selected.
- Refresh again after start/stop actions.
- Avoid constant polling when the user is on a normal provider.

## Error Handling

If start/status/stop fails:

- Keep the UI responsive.
- Show `Ошибка` in the compact card.
- Put the short error in the card tooltip or detail label.
- Do not show long tracebacks in message boxes.

Config snippet failures should show a short warning message and leave the card usable.

## Scope Limits

This feature does not add a full MCP job dashboard yet.

It does not show every connected client or every job in the provider row. The first GUI version is a compact control surface: start, stop, status, and config snippet access.

A later dedicated MCP tab can be added if job history, connected-client visibility, or per-job cancellation needs more space.

## Testing

Add focused tests for `KeyManagementWidget`:

- The combo contains `MCP сервер`.
- Selecting `MCP сервер` hides `keyStatusCard`.
- Selecting `MCP сервер` shows the compact MCP card.
- Switching back to a normal provider hides the MCP card and restores `keyStatusCard`.
- Provider-specific server button remains hidden for MCP mode and still follows existing provider logic outside MCP mode.
- The shared widget exposes the MCP item through the same class used by the main translator, AI glossary generation, AI glossary correction, untranslated fixing, and consistency checking.

Add MCP card logic tests with faked start/status/stop functions so tests do not bind network ports.

Run:

```bash
.venv/bin/python -m pytest tests/test_key_management_widget.py -q
.venv/bin/python -m pytest tests/test_mcp_daemon.py tests/test_mcp_client_install.py -q
```
